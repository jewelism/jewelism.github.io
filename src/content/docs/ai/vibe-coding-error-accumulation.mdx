## 핵심 구조: 왜 오류가 누적되는가

에이전트는 세 가지 이유로 오류를 누적시킨다.

1. **학습 불가** — 같은 실수를 무한히 반복한다
2. **병목 부재** — 인간보다 압도적으로 빠르게 오류를 쌓는다
3. **검색 재현율 하락** — 코드베이스가 커질수록 에이전트가 관련 코드를 찾지 못해 중복과 불일치가 증가한다

결과: 인간 팀이 수년에 걸쳐 만드는 혼란을 에이전트 팀은 **몇 주 만에** 재현한다.

---

## 실무적 대응 (단기)

- **프롬프트를 단일 책임 단위로 분할** — 한 번에 하나만 요청하고 검증
- **코드와 테스트를 함께 생성** — 단, 테스트의 *의도* 자체를 반드시 사람이 리뷰
- **아키텍처 문서를 별도 파일로 관리** — 매 요청마다 컨텍스트로 제공
- **3~4회 생성 후 리팩토링** — "중복 제거", "에러 처리 통일" 등 목표를 명시

한계: 오류를 *관리*할 뿐, 구조적 원인을 해결하지는 못한다.

---

## 전략적 판단 (중기)

**에이전트에게 맡겨도 되는 작업:**
- 범위가 작고 비핵심적인 작업
- 자체 평가 루프가 가능한 작업 (성능 측정, 실험)
- 실패해도 치명적이지 않은 작업

**반드시 사람이 해야 하는 것:**
- 아키텍처와 API 설계
- 하루에 생성되는 코드량을 *검토 가능한 수준*으로 제한
- 최종 품질 게이트

> "자동차에 브레이크가 있어야 빨리 달릴 수 있다" — 인간이 이해할 수 있는 속도로 일해야 한다

---

## 패러다임 전환 (장기): Spec 기반 접근

대화형 코딩의 근본 문제는 **컨텍스트 드리프트**다. 대화가 길어질수록 초기 의도에서 멀어진다.

**Chat → Spec → Code** 흐름:
1. 인간이 요구사항을 자연어로 전달
2. 에이전트가 spec을 작성, **인간이 의도 수준에서 검토**
3. spec만 보고 코드를 자동 생성 — 대화 이력 누적 없음

핵심 효과: 오류 발생 지점을 "코드"에서 "명세"로 끌어올린다. 코드 레벨에서 오류를 잡는 건 이미 복잡성이 폭발한 후지만, spec에서 모순을 잡으면 오류가 코드로 전파되기 전에 차단된다.

## 세 층위 요약

| | 단기 (전술) | 중기 (전략) | 장기 (패러다임) |
|---|---|---|---|
| **핵심 질문** | 오류를 어떻게 줄이나? | 어디까지 맡길 것인가? | 왜 구조적으로 실패하는가? |
| **해결 방향** | 프롬프트 분할·테스트·타입 시스템 | 범위 제한·인간이 품질 게이트 | Spec 기반 패러다임 전환 |
| **인간의 역할** | 테스트 의도 리뷰어 | 아키텍처 작성자 | 명세 감리자 |
